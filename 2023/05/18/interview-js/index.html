对 HTML 语义化标签的理解

  • HTML5 语义化标签是指正确的标签包含了正确的内容，结构良好，便于阅读，比如 nav 表示导航条，类似的还有 article、header、footer 等等标签

Flex：弹性布局（一维）

  1. 基本概念：容器/项目，主轴/交叉轴，起始线/结束线

  2. 容器属性：

    • flex-direction（主轴方向）

      • row（默认值）：主轴为水平方向，起点在左端

      • row-reverse：主轴为水平方向，起点在右端

      • column：主轴为垂直方向，起点在上沿

      • column-reverse：主轴为垂直方向，起点在下沿

    • flex-wrap（换行）

      • nowrap：默认值，不换行
      • wrap：换行，第一行在上方
      • wrap-reverse：换行，第一行在下方

    • flex-flow（direction 和 wrap 的组合）

    • justify-content（定义项目在主轴上对齐的方式）

      • flex-start：左对齐
      • flex-end：右对齐
      • center：居中
      • space-between：两端对齐，项目之间间隔相等
      • spaec-around：每个项目两侧间隔相等

    • align-items（定义项目在交叉轴上对齐的方式）

      • stretch：默认值，如果项目未设置高度或者设为 auto，自动占满整个容器的高度

      • flex-start：交叉轴的起点对齐

      • flex-end：交叉轴的终点对齐

      • center：交叉轴的中心对齐

      • baseline：项目的第一行文字基线对齐

      • align-content（定义了多根轴线的对齐方式）

        • flex-start：与交叉轴的起点对齐

        • flex-end：与交叉轴的终点对齐

        • center：交叉轴的中点对齐

        • space-between：与交叉轴两端对齐，轴线之间的间隔平均分布

        • space-around：每根轴线两侧的间隔都相等。所以，轴线之间的间隔比轴线与边框的间隔大一倍

        • stretch（默认值）：轴线占满整个交叉轴

  3. 项目属性：

    • order：定义项目的排序，数值越小，排列越靠前，默认为 0

    • flex-grow：剩余空间充足时，定义项目放大比例，默认为 0

    • flex-shrick：剩余空间不足时，定义项目缩小比例，默认为 1

    • flex-basis：定义了在分配多余空间之前，项目占据的主轴空间（main size）

    • flex：flex-grow、flex-shrick、flex-basis 的简写

    • align-self：允许单个项目有与其他项目不一样的对齐方式，可覆盖 align-item 属性

script 标签中的 defer 和 async

  1. 问题来源：script 加载会阻塞页面的加载，如果我们引用外部 js，在网络不佳的情况下，就会导致页面长期处于白屏状态。

  2. 解决方法：script 标签 defer、async

    • defer：

      • 不会阻止页面的解析，并下载此脚本，当页面解析完成后，立即执行下载的脚本（若需要下载多个脚本，则等待其他脚本加载完成，再执行脚本），脚本执行完成后才会触发 DOMContentLoaded 事件

    • async：

      • 不阻塞浏览器解析 HTML，但是 script 下载完成后，会立即中断浏览器解析 HTML，并执行此 script
      • 互相独立，谁先下载完，谁先执行，没有固定的先后顺序，不可控

  3. 最大区别在于执行时机

什么是 BFC？BFC 有什么特性？如何创建 BFC？BFC 的应用场景？

  1. BFC 是块级格式化上下文，是在盒模型下对块级元素进行布局的独立渲染区域，该区域内的元素和区域外的元素互不影响
  2. 特性：
    • 同一个 BFC 下 margin 会重叠
    • BFC 内部的元素不会影响外部的元素
    • BFC 是一个块级元素，块级元素在垂直方向上依次排列
    • BFC 区域不会与 float 元素重叠
  3. 如何创建
    • 定位元素：position设置为absolute或者flexed
    • 浮动元素：float不为none
    • overflow：overflow设置为hidden
    • 布局：inline-block、flexed
  4. 应用场景：
    • 防止两个相邻块级元素的上下 margin 值发生重叠（例：上下 margin 重合）
    • 清除浮动（例：父元素没有设置高度，子元素浮动溢出）
    • 实现自适应布局，防止元素被浮动元素覆盖（左边固定，右边自适应）

什么是 DOM 事件流？什么是事件委托？

  1. 事件发生时会在元素节点间按照特定的顺序传播,这个传播过程即 DOM 事件流。
    • dom 事件流分为三个阶段：捕获阶段、目标阶段、冒泡阶段
    • addEventListener (type, listener,[useCapture] )，第三个参数不写默认是 false，false 表示在事件冒泡阶段，如果是true，表示事件在捕获阶段
  2. 事件委托
    • 不给每个子节点单独设置事件监听器，而是给父节点设置事件监听器，利用事件冒泡原理影响设置每个子节点
    • 优点：减少内存消耗，提高性能

什么是事件捕获？什么是事件冒泡？

  1. 事件捕获
    • 自上而下执行
    • 事件捕获用意在于事件到达预定目标之前捕获它
  2. 事件冒泡
    • 自下而上执行
    • 开始由某个具体的事件接受，然后逐级向上传播到最顶层的节点
    • 阻止事件冒泡：e.stopPropagation或者cancelBubble = true

position 的属性

  1. static（默认定位）
    • 默认值，没有定位，元素出现在正常的流中，top, right, bottom, left 和 z-index属性无效
  2. relative（相对定位）
    • 在元素原有的位置基础上，根据设置的top,left调整元素位置
    • 不会改变页面布局，不影响其他元素的偏移，因此会在此元素未添加定位时所在位置留下空白
  3. absolute（绝对定位）
    • 绝对定位的元素的位置相对于最近的已定位父元素，如果元素没有已定位的父元素，那么它的位置相对于
    • 绝对定位使元素的位置与文档流无关，因此不占据空间
    • 绝对定位的元素和其他元素重叠
  4. fixed（固定定位）
    • 元素的位置相对于浏览器窗口是固定位置，即使窗口是滚动的它也不会移动
    • fixed 定位使元素的位置与文档流无关，因此不占据空间
    • fixed 定位的元素和其他元素重叠
  5. sticky（粘性定位）
    • 粘性定位可以被认为是相对定位和固定定位的混合。元素在跨越特定阈值前为相对定位，之后为固定定位。

为什么 0.1+0.2 > 0.3?

  • 因为 js 的底层中，每个变量是以二进制表示，固定长度是 64 位，其中第 1 位是符号位，再往后 15 位是指数位，最后 52 位是尾数位，而 0.1 和 0.2 转成二进制时是无限循环小数，js 进行了截取，所以此时的数值并不是原来的值，会比原来的值大

  • 怎么解决？同时放大倍数，再除以相应的倍数

什么是闭包？闭包的优缺点？闭包的应用？

  1. 闭包是嵌套的内部函数，包含被引用变量（函数）的对象

  • 优点：

    • 可以读取函数内部的变量

    • 将局部变量保存在内存中，实现数据的共享

    • 可以避免使用全局变量，避免全局变量污染

  • 缺点：

    • 被引用的私有变量不能被销毁，增大了内存消耗，滥用闭包可能导致内存泄露，解决方法是可以在使用完变量后手动为它赋值为 null
    • 闭包涉及跨域访问，所以会导致性能损失，我们可以通过把跨作用域变量存储在局部变量中，然后直接访问局部变量，来减轻对执行速度的影响
  1. 应用
    • 模拟私有方法
    • setTimeout 的循环
    • 匿名自执行函数
    • 结果要缓存场景

内存泄露、垃圾回收机制：

  • 什么是内存泄漏？
    • 指不再使用的内存没有被释放，无法使用这段内存
  • 为什么会导致内存泄漏？
    • 指 js 无法访问这个对象，垃圾回收机制却认为该对象还在使用，因此垃圾回收机制不会释放对象，导致该内存不会被释放，越积越多，最终导致系统崩溃。
  • 垃圾回收机制有什么方法？
    • 标记清除法：垃圾回收机制获取根并标记他们，然后访问并标记所有来自它们的引用，然后在访问这些对象并标记它们的引用…如此递进结束后若发现有没有标记的（不可达的）进行删除，进入执行环境的不能进行删除

单线程

  • 为什么 js 是单线程的？
    • 因为 js 里面有可视的 dom，如果是多线程的话，这个线程正在删除 dom 节点，另一个线程正在编辑 dom 节点，导致浏览器不知道处理哪个线程。

同步异步

  • js 任务分为同步和异步任务，异步任务又分成宏任务和微任务。
    • 宏任务：script、setTimeout、setInterval、setImmediate、i/o 操作（输入输出，比如读取、文件操作、网络请求）、ui render（dom 渲染，即更改代码重新渲染 dom 的过程）、异步 ajax 等
    • 微任务：Promise（then、catch、finally）、async/await、process.nextTick、Object.observe(⽤来实时监测 js 中对象的变化)、 MutationObserver(监听 DOM 树的变化)
  • 执行顺序
    • js 代码在执行的时候，会先执行同步代码
    • 执行宏任务 script，
    • 进入 script 后，所有的同步任务主线程执行
    • 所有宏任务放入宏任务执行队列
    • 所有微任务放入微任务执行队列
    • 先清空微任务队列，
    • 再取一个宏任务，执行，再清空微任务队列
    • 依次循环

有使用过 Vue 吗？说说你对 Vue 的理解

说说 vue 的优缺点

描述 vue 生命周期

  1. 什么是 vue 的生命周期？
    • vue 从实例的创建到销毁的过程，就是生命周期，整体上分为创建，挂载，更新，销毁
  2. 细分，每一步分两小步
    • beforCreate
      • 创建之前，data 和 method 还没有
      • 在 beforCreate 和 created 挂载 data 和绑定事件
    • created
      • 创建完成，data 和 method 可以用了
      • 在 created 和 beforMount 之间，如果没有设置 el 则生命周期结束，如果有则继续，el 挂载结束后，根据 templete/outerHTML(el)渲染页面
    • beforMount
      • 在 beforMount 之前已创建虚拟 dom
      • 之后在 mounted 之前，将 vm.$el 替换掉页面中的 el
    • mounted
      • 页面渲染完成
      • 一个生命周期直挂载一次
    • beforUpdate
      • data 变化后，对应组件重新渲染之前
    • updated
      • data 变化后，对应组件重新渲染完成
    • beforDestroy
      • 在实例销毁之前，实例仍然可用，手动撤销监听事件，计时器等
    • destroyed
      • 仅存在 Dom 节点，其他所有东西已自动销毁

vue 双向绑定原理

  • vue 数据双向绑定通过数据劫持结合发布订阅模式，利用 Object.defineProperty()来劫持各个属性的 getter、setter，在数据更新变动时发布消息给订阅者，触发响应的监听回调

MVVM 的原理

  • MVVM 作为绑定的入口，整合 Observer、Compile、Watcher 三者，通过 Observer 来监听数据变化，Compile 解析模板指令，Watcher 建立起 Observer 和 Compile 之间的联系，达到数据更新使得视图更新，视图更新使得数据更新的双向绑定

MVC

  • M：模型，负责处理数据逻辑的部分
  • V：视图，负责处理数据展示的部分
  • C：控制器，负责处理用户交互的部分

vue 的发布订阅模式

  • data 中每个数据绑定一个发布者 Dep，这个 Dep 收集了依赖该数据的所有订阅者

  • 在数据变化后，触发相应 setter，从而 Dep 发布通知，通知每个订阅者，做出对应的回调函数

Vue 如何监听数组

  • vue 重写了数组的 push、pop、shift、unshift、splice、sort、reverse 七种方法，调用这些方法后做出回调更新

Vue 为什么采用异步更新？

  • 因为 vue 是组件级的更新，在更新非常频繁的情况下，如果不采用异步更新的话每次都需要重新渲染，非常浪费性能
  • 每次数据更新的时候，vue 会把它放到一个队列中，等到最后会调用 nextTick 方法，nextTick 会清空这个队列
  • nextTick(callback)也可以手动调用，同样会把 callback 回调函数放到队列的最后，保证视图更新完成再被调用

Vue 中的 nextTick

  • 解释：在下次 Dom 更新循环结束之后执行延迟回调，在数据修改之后立即执行使用这个方法，获取更新后的 Dom
  • 应用：
    • 想要在生命周期函数 create()中操作 Dom
    • 在数据改变之后执行的操作，而这个操作需要在 Dom 结构改变后才操作
    • 在使用某个第三方插件时 ，希望在 vue 生成的某些 dom 动态发生变化时重新应用该插件

computed、methods、watch 的区别

  1. computed 是计算属性，基于它的依赖进行缓存，只有依赖发生改变时才会重新求值

  2. methods 是方法，表示一个具体的操作，主要用来书写业务逻辑

  3. watch 用来监听数据的变化，从而进行某些业务逻辑的操作，可以看作是 computed 和 methods 的结合体

  4. 应用：

    • 数据量大，需要缓存的时候用 computed ；每次确实需要重新加载，不需要缓存时用 methods

    • 当我们需要进行数值计算时，并且依赖于其他数据时，应该使用 computed；当我们需要在数据变化时执行异步或者开销较大的操作时，应该使用 watch

什么是虚拟 dom?

  • 虚拟 dom 是相对于浏览器所渲染出来的真实 dom 的，它用 js 对象描述 dom 结构。

虚拟 Dom 的优缺点

  1. 优点
    • 减少 dom 操作，减少 dom 回流和重绘
    • 保证性能的下限，虽说性能不是最佳，但是它具备局部更新的能力，所以大部分时候还是比正常的 DOM 性能高很多的
  2. 缺点
    • 首次渲染大量 DOM 时，由于多了一层虚拟 DOM 的计算，会比 innerHTML 插入慢

虚拟 Dom 生成的过程

  • 首先当生命周期走到 created 和 beforMount 之间时，会把 template 模板编译成 render 函数，然后在 render 函数运行时调用 h 函数，h 函数内部调用 vnode 函数生成虚拟 dom，虚拟 dom 首次生成
  • 之后，当数据变化时生成新的虚拟 dom，新旧虚拟 dom 用 diff 算法进行比较

diff 算法

  • diff 算法是指对新旧虚拟节点进行对比，并返回一个 patch 对象，用来存储两个节点不同的地方，最后利用 patch 记录的消息局部更新 DOM
  • 步骤：
    • 比较两个虚拟 dom 树，对根节点 root 进行执行 patch(oldVnode，newVnode)函数，比较两个根节点是否是相同节点。如果不同，直接替换（新增新的，删除旧的）
    • 如果相同，对两个节点执行 patchVnode（oldVnode, newVnode），比较属性，文本，以及子节点。此时，要么新增，要么删除，要么直接修改文本内容。只有当都存在子节点时，并且 oldVnode === newVnode 为 false 时。会执行 updateChildren 函数，去进一步比较他们的子节点。
    • 分情况比较
    • 递归比较子节点

v-if 和 v-show 有什么区别?

  • 都是用来显示或隐藏元素
  • v-if 是根据条件完全销毁或重建元素，适用对性能要求高的
  • v-show 是修改 display 样式来显示或隐藏，适用频繁的显示和隐藏

vue 中 key 的作用

  • key 主要用在虚拟 Dom 算法中，每个虚拟节点 VNode 有一个唯一标识 Key，通过对比新旧节点的 key 来判断节点是否改变，用 key 就可以大大提高渲染效率

v-for 为什么要用 key？

  • vue 中列表渲染需添加 key=’唯一标识’，因为 vue 组件高度复用，用 key 可以标识组件的唯一性，主要是为了高效的更新虚拟 dom
  • key 主要是用来做 diff 算法，diff 是同级比较，比较新旧标签上的 key 和标签名，如果都一样只是移动处理，并不会重新创建或删除元素
  • 没有 key 时默认采用的是“就地复用”的策略，如果数据项的顺序发生变化，vue 不是移动 dom 元素来匹配数据项的改变，而是简单的复用原来的位置的每个元素。如果删除第一个元素，在进行比较时发现标签一样但值不一样时，就会复用之前的位置，将新值直接放到该位置，以此类推，最后多出一个就会把最后一个删除掉

为什么组件中的 data 为什么是个函数？

  • 组件中的 data 是一个函数，是因为每个组件都是独立的实例，如果 data 是一个对象，则所有的实例共享同一个 data 对象，就会导致数据混乱。而将 data 定义成一个函数，每个实例都会调用该函数，返回一个独立的 data 对象，保证数据的独立性

vue3 相比于 vue2 有什么升级

  1. Composition API（组合 api）

    • Vue2 使用的是 options 的 API ，代码逻辑比较分散，可读性差，可维护性差
    • Vue3 以功能为单位的代码组织方式，同时它可以让代码更易重用，逻辑分明，可维护性高

  2. 自定义渲染 api（Custom Renderer API），只需重写createApp，定义自己的 render 渲染生成函数

  3. 更好的支持 typescript

  4. 新的组件

    • Fragment

      • vue2 每个组件都需要一个根节点

      • vue3 允许没有根组件，自动创建一个虚拟节点的Fragment，不会出现在 dom 树

    • Suspense

      • Suspense 能够暂停你的组件渲染，并渲染后备组件，直到条件满足为止，进行异步操作的组件

  5. 更快的开发体验，vite 构建工具

    • 利用浏览器去解析 imports，在服务器端按需编译返回，完全跳过了打包这个概念，服务器随起随用
    • webpack 在随着项目的增大加载变慢，热重载有延迟，但 vite 热重载更快

  6. 按需编译，体积比 vue2.x 更小，tree shaking，可以将无用模块“剪枝”

  7. diff 算法优化：

    • Vue2.x 中的虚拟 dom 是进行全量的对比

    • Vue3.0 中新增了静态标记PatchFlag，在与上次虚拟结点进行对比的时候，只对比带有patch flag的节点

  8. 响应式原理 Proxy

    • Vue2 的响应式原理是利用Object.defineProperty()数据劫持，结合发布订阅模式的方式来实现的
    • Vue3 中使用了 es6 的Proxy对数据代理

Vue3.0 里为什么要用 Proxy API 替代 defineProperty API？

  • defineProperty API 的局限性最大原因是它只能针对单例属性做监听，无法监听通过索引修改属性的值的变化或者新增一个预先不存在的值的变化
  • Proxy API 的监听是针对一个对象的，那么对这个对象的所有操作会进入监听操作，这就完全可以代理所有属性，将会带来很大的性能提升和更优的代码
  • defineProperty 执行 Object.defineProperty 把每一层对象数据都变成响应式的，这无疑会有很大的性能消耗，而 proxy 按需实现响应式，减少性能消耗

vue 开发中的性能优化

  1. 不做响应式：对于不改变的数据，使用Object.freeze()冻结 data 中的属性，从而阻止数据劫持，提升渲染速度
  2. 图片懒加载：第三方插件vue-lazyload

  4. 事件的销毁：组件销毁时，会自动解绑它的全部指令及监听器，但仅限于组件本身的事件，而对于定时器、addEventListener注册需要在组件销毁的生命周期钩子中手动销毁或解绑，以免内存泄漏
  5. 第三方插件按需加载
  6. 使用 keep-alive 缓存页面
  7. 在使用 for 的同时避免使用 v-if，vue2 中 for 的优先级比 v-if 高
  8. 使用 v-show 复用 dom

什么是跨域

  • 跨域是指浏览器不能执行其他网站的脚本。它是浏览器同源策略造成的，是浏览器对 JS 实施的安全限制

什么是同源

  • 指发出请求的页面的协议、域名、端口与被请求的页面相同

什么是同源策略

  • 一种约定，它是浏览器最核心也最基本的安全功能，如果缺少了同源策略，则浏览器的正常功能可能都会受到影响

解决跨域问题

  1. JSONP：利用的是 script 标签 src 属性请求 js 无跨域问题，但具有局限性，只能发送 get 请求

  2. CORS：

    • 它允许浏览器向跨源服务器，发出 XMLHttpRequest 请求，从而克服了 AJAX 只能同源使用的限制

    • 普通跨域请求：只服务端设置 Access-Control-Allow-Origin 即可，前端无须设置，若要带 cookie 请求：前后端都需要设置

  3. Nginx 反向代理接口跨域：可以将前端项目部署到和接口同源的当前本地的服务器上

  4. web sockets

http 的报文结构

  1. 请求报文
    • 请求行、请求头、空体、请求体
      • 请求行：请求方法、请求 URL、http 协议版本
  2. 响应报文
    • 状态行、消息报头、空行、响应正文

http 报文头部有哪些字段

  • 请求头（request headers）
    • cache-control 是否使用缓存
    • Connection：keep-alive 与服务器的连接状态
    • Host 主机域
  • 返回头（response headers）
    • cache-control
    • etag 唯一标识，缓存用的

  • 2XX：成功，操作被成功接收并处理

  • 3XX：重定向，需要进一步的操作以完成请求

    • 302 临时重定向
    • 303 表示可以在缓存中取数据（协商缓存）

  • 5XX：服务端错误

http 和 https 有什么区别

  • https 协议需要CA证书，费用较高

  • http 是超文本传输协议，信息是明文传输；https 则是具有安全性的ssl加密传输协议

  • 使用不同的链接方式，端口也不同，一般而言，http 协议的端口为80，https 的端口为443

  • http 的连接很简单，是无状态的；https 协议是由 SSL+ HTTP协议构建的可进行加密传输、身份认证的网络协议，比 http 协议安全

https 的实现原理？

  • 客户使用 https url 访问服务器，则要求 web 服务器建立 ssl 链接

  • web 服务器接收到客户端的请求之后，会将网站的证书（证书中包含了公钥）传输给客户端

  • 客户端和 web 服务器端开始协商 SSL 链接的安全等级，也就是加密等级

  • 客户端浏览器通过双方协商一致的安全等级，建立会话密钥，然后通过网站的公钥来加密会话密钥，并传送给网站

  • web 服务器通过自己的私钥解密出会话密钥

  • web 服务器通过会话密钥加密与客户端之间的通信

http1.0 和 http1.1，还有 http2 有什么区别？

  • http0.9 只支持 GET 请求
  • http1.0 增加了 POST，HEAD，OPTION，PUT，DELETE 等
  • http1.1 增加了长连接 keep-alive、增加了 host 域、更节约带宽了、还有缓存中的 max-age（之前是 expire）
  • http2.0 增加了服务器推送、多路复用、头部压缩、以及更接近二进制了
  • http3.0 增加了 QUIC 协议，是基于 udp 来完成建立连接的

localStorage、SessionStorage、cookie、session 之间有什么区别

  • localStorage
    • 生命周期：关闭浏览器后数据依然保留，除非手动清除，否则一直在
    • 作用域：相同浏览器的不同标签在同源情况下可以共享 localStorage
  • sessionStorage
    • 生命周期：关闭浏览器或者标签后即失效
    • 作用域：只在当前标签可用，当前标签的 iframe 中且同源可以共享
  • cookie、session 的区别
    • 保存方式
      • Cookie 保存在浏览器端，Session 保存在服务器端
    • 使用机制
    • 存储内容
      • Cookie 只能保存字符串类型，以文本的方式
      • Session 通过类似 Hashtable 的数据结构来保存，能支持任何类型的对象(Session 中可含有多个对象)
    • 存储的大小
      • 单个 Cookie 保存的数据不能超过 4kb
      • Session 大小没有限制
    • 安全性
      • Cookie 存在的攻击 Cookie 欺骗，Cookie 截获
      • Session 的安全性大于 Cookie

TCP 和 UDP 的区别

  1. TCP 是面向连接的；UDP 是无连接的即发送数据前不需要先建立链接。
  2. TCP 提供可靠的服务。也就是说，通过 TCP 连接传送的数据，无差错，不丢失，不重复且按序到达，适合大数据量的交换；UDP 尽最大努力交付，即不保证可靠交付。
  3. TCP 是面向字节流；UDP 面向报文，并且网络出现拥塞不会使得发送速率降低（因此会出现丢包，对实时的应用比如 IP 电话和视频会议等）
  4. TCP 只能是 1 对 1 的，UDP 支持 1 对 1，1 对多，多对多
  5. TCP 的首部较大为 20 字节，而 UDP 只有 8 字节

get 和 post 的区别

  • 传参
    • get 参数通过 url 传递，post 放在 request body 中
  • 参数长度
    • get 请求在 url 中传递的参数是有长度限制的，而 post 没有
  • 安全性
    • get 比 post 更不安全，因为参数直接暴露在 url 中，所以不能用来传递敏感信息
  • 字符编码
    • get 请求只能进行 url 编码，而 post 支持多种编码方式

TCP 三次握手四次挥手

  • 客户端与用户端三次握手建立连接，四次挥手释放连接
  1. 三次握手过程：客户端向服务端发送 SYN 包，进入 SYN_SENT 状态，服务端接收到 SYN 后，向客户端返回 SYN+ACK 包，表示 SYN 已收到，进入 SYN_RECEIVE 状态，最后客户端向服务端发送 ACK 包表示确认，双方进入 ESTABLISH 状态
    • 为什么不是两次握手？
      • 之所以是三次握手而不是两次，是因为如果只有两次，在服务端收到 SYN 后，向客户端返回一个 ACK 确认就进入 establish 状态，万一这个请求中间遇到网络情况而没有传给客户端，客户端一直是等待状态，后面服务端发送的信息客户端也接受不到了
    • 三次握手可以携带数据吗？
      • 第一次、第二次握手不可以携带数据，而第三次握手是可以携带数据的。假设第一次可以携带数据，如果有人恶意攻击服务器，每次都在第一次握手中的 SYN 报文放入大量数据，重复发送大量 SYN 报文，此时服务器会花费大量内存空间来缓冲这些报文，服务器就更容易被攻击了
  2. 四次挥手过程：首先客户端向服务端发送一个 FIN 包，进入 FIN_WAIT1 状态，服务端收到后，向客户端发送 ACK 确认包，进 CLOSE_WAIT 状态，然后客户端收到 ACK 包后进入 FIN_WAIT2 状态，然后服务端再把自己剩余没传完的数据发送给客户端，发送完毕后在发送一个 FIN+ACK 包，进入 LAST_ACK（最后确认）状态，客户端收到 FIN+ACK 包后，再向服务端发送 ACK 包，在等待两个周期后在关闭连接

什么是 token 认证？

  • 在客户端输入账号密码后，向服务端发送请求，服务端返回 token 给客户端，下次客户端发送请求时请求头携带该 token，服务端收到后，会与之前保存的 token 对比验证，验证通过返回数据，否则不返回

什么是 JWT（Json Web Token）？

  • 一个开放标准(RFC 7519)，它定义了一种紧凑的、自包含的方式，用于作为 JSON 对象在各方之间安全地传输信息
  • 过程：
    • 用户输入账号密码后，向服务端发送请求
    • 服务端进行用户验证，然后创建 JWT 字符串返回给客户端
    • 客户端请求接口时，在请求头携带 JWT
    • 服务端验证 JWT 合法性，如果合法则继续调用并返回数据
  • 和 token 的区别在于 token 存储在客户端

讲讲 http 缓存

  • 缓存分为强缓存和协商缓存
    • 强缓存
      • 在客户端发送请求之前，先检查强缓存，查看强缓存的 cache-control 里的 max-age，如果 max-age 没有过期，直接使用该缓存
    • 协商缓存
      • 浏览器加载资源时，没有命中强缓存，这时候就去请求服务器，去请求服务器的时候，会带着两个参数，一个是 If-None-Match，也就是响应头中的 etag 属性，每个文件对应一个 etag；另一个参数是 If-Modified-Since，也就是响应头中的 Last-Modified 属性，带着这两个参数去检验缓存是否真的过期，如果没有过期，则服务器会给浏览器返回一个 304 状态码，表示缓存没有过期，可以使用旧缓存

从浏览器输入 url 后经历了什么

  1. DNS 解析
    • 首先，浏览器会进行 DNS 解析，将 URL 中包含的域名解析为对应的 IP 地址。它会检查浏览器缓存、操作系统缓存和路由器缓存，如果没有找到，那么就会去互联网服务提供商（ISP）的 DNS 服务器上寻找。如果 DNS 解析成功，它将获得对应的 IP 地址
  2. TCP 连接
    • 接下来，浏览器会通过 TCP/IP 协议与 Web 服务器建立 TCP 连接。这个过程需要进行三次握手。在三次握手完成后，浏览器和服务器之间就建立起了一条可靠的数据传输通道
  3. 发送 HTTP 请求
    • TCP 连接建立成功后，浏览器便向 Web 服务器发送一个 HTTP 请求，请求资源
  4. 服务器处理请求并返回响应
    • 服务器收到浏览器发送的 HTTP 请求后，开始根据请求的内容生成响应，并把响应返回给浏览器。服务器返回的内容可能是 HTML 页面、CSS 文件、JavaScript 文件、图片或其他资源
  5. 浏览器渲染页面
    • 浏览器接收到服务器返回的响应内容后，开始解析 HTML 代码，构建 DOM 树，并将 DOM 和 CSSOM 合成为渲染树。然后，浏览器使用渲染树布局来确定每个节点在屏幕上的精确位置，并绘制这些节点以显示最终图像
  6. 断开连接
    • 当浏览器获得服务器响应后，在传输完成后，浏览器和服务器之间的 TCP 连接会被断开

crfs 攻击原理

  • 跨站请求伪造，用户登录了 A 页在不退出登录的情况下，访问了危险网站 B 页，这个时候 B 页带着 A 页的 cookie 向 A 的服务端发起请求，会让服务端认为这个是可信任用户，从而达到攻击的目的
  • 防御方式，使用 token

xss 攻击原理

  • 跨站脚本攻击，一般是指黑客通过在网页中注入恶意脚本，当用户浏览网页时，恶意脚本执行，控制用户浏览器行为的一种攻击方式
